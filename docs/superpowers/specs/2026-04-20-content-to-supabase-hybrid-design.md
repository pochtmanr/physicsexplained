# Content Architecture — Hybrid (MDX source → Supabase blocks + translations)

**Date:** 2026-04-20
**Status:** proposed
**Supersedes:** nothing (first formal spec for content storage beyond MDX)
**Repo:** `/Users/romanpochtman/Developer/physics`
**Supabase project:** `cpcgkkedcfbnlfpzutrc` (`physics.explained`, eu-west-1)

## Goal

Move long-form content out of per-locale MDX files and into Supabase as structured blocks, so the same data feeds the web app, future mobile apps, and AI retrieval — without losing the MDX authoring ergonomics for the source (English) version.

## Why now

Three forces push at once:

1. **Language scaling.** Going from 2 → 5+ locales with the current model means 155+ MDX files plus parallel English-only TS metadata. Translators have to author through PRs.
2. **Mobile clients.** Swift/Kotlin cannot consume MDX. A neutral block format is needed regardless.
3. **AI retrieval.** The planned photo-to-lesson feature needs embeddings + semantic search over structured chunks. Flat files can't do this.

All three are solved by the same move: publish structured blocks + translations into Supabase once, let every client read from there.

## Non-goals (explicit)

- **Not** a headless CMS with an editor UI. English authoring stays in MDX in the repo.
- **Not** replacing UI string translations (`messages/{locale}/*.json`). Those stay in next-intl as-is.
- **Not** moving slugs, branch lists, or structural identifiers into the DB. Those are code.
- **Not** moving React simulation components (e.g. `pendulum-playground`) into the DB. They stay in `components/`; blocks reference them by name.
- **Not** building the AI photo-to-lesson feature in this spec. This spec prepares the substrate; the AI feature is a later spec.
- **Not** building an admin UI for reviewing translations. Supabase Studio is enough for v1.

## Model: what lives where

| Layer | Source of truth | Consumed by |
|---|---|---|
| UI strings (nav, home, footer, legal, 404) | `messages/{locale}/*.json` | web only (next-intl) |
| Slugs + structural graph (branches, topic slugs, physicist slugs, glossary slugs, relatedTopics edges) | `lib/content/*.ts` | web, mobile, AI (shipped as JSON) |
| **Long-form content blocks** (topic body, physicist bio/contributions, glossary description/history) | **Supabase** — English row generated from MDX by build pipeline; other locales generated by translation pipeline | web, mobile, AI |
| Physics simulations, math utilities | `lib/physics/`, `components/` (React) | web (mobile reimplements natively if needed) |
| Images (physicist portraits, figures) | Supabase Storage bucket `images` (already set up) | all clients |
| Embeddings for AI retrieval | Supabase `content_embeddings` (pgvector) | AI endpoints |

## Block schema

One shared schema across topics, physicists, and glossary. Stored as JSONB. A content row is an ordered array of blocks.

Grounded in what today's MDX actually uses: `<Section index title>`, `<EquationBlock id>`, `<SceneCard caption>` with either images or interactive components (`<PendulumScene>`, `<PhasePortrait>`), `<Callout variant>`, and paragraphs with `<PhysicistLink>` / `<Term>` inline.

```ts
type Block =
  // A numbered article section. Contains other blocks. Only appears at top level.
  | { type: "section"; index: number; title: string; children: Block[] }
  // Headings below section level (for sub-structure inside a section).
  | { type: "heading"; level: 3 | 4; text: string }
  | { type: "paragraph"; inlines: Inline[] }
  // Numbered display equation, addressable by id (e.g. "EQ.01").
  | { type: "equation"; id?: string; tex: string }
  // Figure = caption + one "content" element: image OR registered simulation component.
  | { type: "figure"; caption?: string; content: FigureContent }
  | { type: "callout"; variant: "math" | "history" | "insight" | "aside"; children: Block[] }
  | { type: "list"; ordered: boolean; items: Inline[][] };

type FigureContent =
  | { kind: "image"; src: string; alt: string }    // src is a path inside the `images` bucket
  | { kind: "simulation"; component: string; props?: Record<string, unknown> };

type Inline =
  | string
  | { kind: "em"; text: string }
  | { kind: "strong"; text: string }
  | { kind: "code"; text: string }
  | { kind: "formula"; tex: string }                        // inline KaTeX
  | { kind: "link"; href: string; text: string }
  | { kind: "term"; slug: string; text?: string }           // dotted-underline glossary link
  | { kind: "physicist"; slug: string; text?: string };     // cyan-underline physicist link
```

**Invariants**
- `section` blocks only appear at the top level of a topic's `blocks` array — they do not nest.
- `slug` values on `term` / `physicist` inlines must resolve at build/publish time (fail loudly).
- `FigureContent.simulation.component` must be in the hand-maintained allowlist (`lib/content/simulation-registry.ts`) — prevents arbitrary component references from a DB row.
- `image.src` is a path within the `images` Supabase bucket (e.g. `"figures/kepler-orbits.webp"`); renderer composes the public URL via the existing `storageUrl()` helper.
- `equation.id` is free-form (today: `"EQ.01"`, `"EQ.02"`) — used for in-page cross-references.

## Page chrome (stays out of blocks)

The MDX files today wrap their body in `<TopicPageLayout aside={[...]}>` + `<TopicHeader title subtitle eyebrow>`. That's page chrome, not content. It stays in `page.tsx`:

- `title`, `subtitle`, and any eyebrow text → DB columns (`title`, `subtitle`, and `meta.eyebrow` if needed).
- Sidebar aside content → a second JSONB column `aside_blocks` on `content_entries`. Renderer is the same `<ContentBlocks>` component pointed at a different array.
- Sections / equations / callouts / figures / body prose → the main `blocks` array.

This keeps the DB row focused on translatable content and leaves visual chrome (grid, back-link, section-number pill style) owned by React.

## Database schema

New migration `0002_content.sql`. RLS on, public read, writes service-role only.

```sql
-- Three content kinds, same shape. kind differentiates for polymorphic fetch.
create table public.content_entries (
  kind         text not null check (kind in ('topic', 'physicist', 'glossary')),
  slug         text not null,                    -- topic uses 'branch/topic' form, others use bare slug
  locale       text not null,                    -- 'en', 'he', 'ru', ...
  title        text not null,                    -- rendered as h1 or card title
  subtitle     text,                             -- oneLiner / shortDefinition
  blocks       jsonb not null default '[]',      -- Block[] — main body
  aside_blocks jsonb not null default '[]',      -- Block[] — sidebar content (topics only; empty for physicists/glossary)
  meta         jsonb not null default '{}',      -- kind-specific: { born, died, nationality } for physicist; { category } for glossary; { eyebrow } optional
  source_hash  text,                             -- sha256 of English MDX that produced this row; null for AI-generated locales
  updated_at   timestamptz not null default now(),
  primary key (kind, slug, locale)
);

create index content_entries_kind_locale_idx on public.content_entries (kind, locale);

alter table public.content_entries enable row level security;

create policy "public can read content"
  on public.content_entries for select to anon using (true);

-- Writes: service role only (migration pipeline, translation pipeline).
```

```sql
-- Embeddings table — scaffold only in this spec; populated by a later feature.
create extension if not exists vector;

create table public.content_embeddings (
  id          bigserial primary key,
  kind        text   not null,
  slug        text   not null,
  locale      text   not null,
  chunk_index int    not null,
  chunk_text  text   not null,
  embedding   vector(1536),
  foreign key (kind, slug, locale) references public.content_entries (kind, slug, locale) on delete cascade
);

create index content_embeddings_vec_idx
  on public.content_embeddings using ivfflat (embedding vector_cosine_ops);

alter table public.content_embeddings enable row level security;
create policy "public can read embeddings"
  on public.content_embeddings for select to anon using (true);
```

## Publish pipeline (English source → DB)

A single Node CLI: `pnpm content:publish`.

1. Walk `app/[locale]/(topics)/**/content.en.mdx`, `lib/content/physicists.ts`, `lib/content/glossary.ts`.
2. Parse MDX via `@mdx-js/mdx` + remark/rehype AST into `Block[]`. Custom transformers:
   - `<TopicPageLayout>` → page chrome, dropped. Its `aside={[...]}` prop content → `aside_blocks`.
   - `<TopicHeader title subtitle eyebrow>` → extracted into row columns (`title`, `subtitle`, `meta.eyebrow`). Not a block.
   - `<Section index title>…</Section>` → `{ type: "section", index, title, children: [...] }`
   - `<EquationBlock id>$$…$$</EquationBlock>` (or plain `$$…$$`) → `{ type: "equation", id, tex }`
   - `<SceneCard caption><img … /></SceneCard>` → `{ type: "figure", caption, content: { kind: "image", src, alt } }`
   - `<SceneCard caption><PendulumScene {...props} /></SceneCard>` → `{ type: "figure", caption, content: { kind: "simulation", component: "PendulumScene", props } }`
   - `<Callout variant>` → `{ type: "callout", variant, children: [...] }`
   - `<PhysicistLink slug>` → `{ kind: "physicist", slug, text }` inline
   - `<Term slug>` → `{ kind: "term", slug, text }` inline
   - Inline `$…$` → `{ kind: "formula", tex }` inline
   - Any other JSX tag not listed above → fail the run (opt-in allowlist).
3. Validate every cross-ref slug resolves. Fail the run on unknown refs.
4. Compute `source_hash = sha256(mdx_bytes)`. Upsert one row per (`kind`, `slug`, `locale='en'`).
5. Report diff (added / updated-by-hash / unchanged / removed).

Runs in CI on merge to main, and locally via `pnpm content:publish --dry-run`.

## Translation pipeline (English → other locales)

Separate CLI: `pnpm content:translate --locale he|ru|… [--only topic/kepler]`.

1. For each English row in `content_entries` where no matching locale row exists (or source_hash changed since last translation), emit a translation job.
2. For each job: send the row (title, subtitle, meta.eyebrow, blocks, aside_blocks) to an LLM with a system prompt that lists translatable vs frozen fields.

   **Translate:** `title`, `subtitle`, `meta.eyebrow`, `section.title`, `heading.text`, `figure.caption`, `figure.content.image.alt`, inline bare strings, `em.text` / `strong.text` / `link.text`, `term.text` / `physicist.text` (display labels only), `list.items[*]` content. Recurse into `callout.children` and `section.children`.

   **Never translate:** any `slug`, `tex` (equations are universal), `equation.id`, `href`, `code.text` (code stays in source language), `component` names, simulation `props` (keys and values), `figure.content.image.src`.

   Uses the house-style glossary from `mcp__physics-mcp__get_terminology` per target locale so domain terms render consistently.
3. Validate output: identical block-tree shape (same types, same children counts, same slugs), only text fields differ. Fail the job on structural drift.
4. Upsert locale row with `source_hash` matching the English source at translation time (so we know when a re-translation is needed).

Runs manually after a publish. Cost check: ~31 topics × ~2k words × 4 new locales ≈ 250k tokens ≈ $1–3 with Haiku. Trivial.

Optional follow-up (not this spec): a Supabase view `pending_translations` that lists `(kind, slug, locale)` where source_hash differs from the English row, driven from Studio.

## Edit workflow (day-to-day)

The authoring loop you'll actually use:

```
1. Edit app/[locale]/(topics)/<branch>/<topic>/content.en.mdx
   (or content/physicists/<slug>.en.mdx, content/glossary/<slug>.en.mdx)

2. pnpm content:publish
   → parses the MDX, upserts the English DB row, bumps source_hash.

3. pnpm content:translate --locale he,ru,es,ar …
   → detects source_hash drift on affected rows
   → regenerates those locale rows via LLM, re-upserts.

4. Commit the MDX change. Next deploy (or ISR) picks up the new DB rows.
```

Edits never happen directly in the DB for English — the MDX file is canonical. Translations never go back to MDX — the DB is canonical for non-English locales.

Optionally exposed via the existing `physics-mcp` server as MCP tools (`publish_topic`, `translate_topic_all_locales`) so the loop can be driven by conversation rather than terminal.

## Runtime: how pages fetch content

Pages become thin. Three page files change:

- `app/[locale]/(topics)/[branch]/[topic]/page.tsx`
- `app/[locale]/physicists/[slug]/page.tsx`
- `app/[locale]/dictionary/[slug]/page.tsx`

Each follows the same pattern:

```ts
export async function generateStaticParams() {
  // unchanged — still driven from lib/content/*.ts
}

export default async function Page({ params: { locale, /* slug parts */ } }) {
  const entry = await getContentEntry(kind, slug, locale);
  if (!entry) notFound();
  return <ContentBlocks blocks={entry.blocks} title={entry.title} subtitle={entry.subtitle} />;
}
```

`getContentEntry()` is a server-only helper in `lib/content/fetch.ts` that reads from Supabase with a per-request cache. Called inside RSC — fully static when pages are pre-rendered.

**Locale fallback:** if a locale row doesn't exist for a given `(kind, slug)`, fall back to English and render a small `<LocaleFallbackNotice />` banner at the top. This keeps the site usable during progressive translation rollout.

## `<ContentBlocks>` renderer

New component at `components/content/content-blocks.tsx`.

- Pure switch on `block.type`. `section` and `callout` recurse into children.
- Inline renderer is a recursive walk over `Inline[]`.
- `physicist` / `term` inlines reuse the existing `<PhysicistLink>` / `<Term>` components (which already throw on unknown slugs — keep that behaviour).
- `equation` renders the existing `<EquationBlock>` with `id` and KaTeX-compiled `tex`.
- `figure` with `content.kind === "image"` renders existing `<SceneCard>` around a `next/image`; with `content.kind === "simulation"` looks up `content.component` in `lib/content/simulation-registry.ts` and renders with `content.props`. Unknown component names throw at render.
- `section` renders the existing `<Section>` wrapper with `index` + `title`, body blocks inside.

Input is the same shape whether it came from the DB or a hand-built fixture — all rendering paths share the same component, so unit tests exercise the real render.

The current MDX pages stay in the repo as the authoring source for English. The publish pipeline reads from them; the runtime does not.

## Metadata arrays (the today-is-English-only bug)

`lib/content/physicists.ts` and `lib/content/glossary.ts` currently hold localised strings (`name`, `oneLiner`, `shortDefinition`, `description`, `history`, `bio`, `contributions`) inline in TypeScript. These are English prose, not structural data — they belong in MDX alongside topic content so the authoring flow is uniform.

**One-time migration:**

- Convert each physicist TS entry → `content/physicists/{slug}.en.mdx` with a `<PhysicistHeader name born died nationality oneLiner />` component + bio paragraphs + `<ContributionsList>` block.
- Convert each glossary term → `content/glossary/{slug}.en.mdx` with `<TermHeader term category shortDefinition />` + description + optional history section.
- The publish pipeline treats all three content kinds uniformly — one parser, one code path.
- The TS files shrink to structural data only:
  - Physicist: `slug`, `born`, `died`, `nationality` (locale-invariant facts kept in TS so `meta` can be derived at publish time), `relatedTopics`.
  - Glossary: `slug`, `category`, `relatedPhysicists`, `relatedTopics`.

Card tiles and index grids read `title` + `subtitle` from the DB row for the current locale.

**Alternative (smaller PR):** keep physicist/glossary English copy in TS; bootstrap DB from TS once, and treat the DB rows as source-of-truth thereafter (future English edits happen in DB, not TS). This trades authoring-pipeline uniformity for a smaller migration. Flag if you'd prefer this route.

## Build order

1. **Migration.** `0002_content.sql` — `content_entries` + RLS + `vector` extension + `content_embeddings` scaffold.
2. **Block types + renderer + simulation registry.** `lib/content/blocks.ts` (types), `components/content/content-blocks.tsx` (renderer), `lib/content/simulation-registry.ts` (allowlist currently mapping `"PendulumScene"` and `"PhasePortrait"` to their React components). Unit-tested with hand-written block fixtures; no DB calls yet.
3. **MDX → blocks parser.** `scripts/content/parse-mdx.ts`. Fixture-based tests per JSX tag (`<Section>`, `<EquationBlock>`, `<SceneCard>` with image vs simulation, `<Callout>`, `<PhysicistLink>`, `<Term>`, inline `$…$`). Plus golden-file tests on the five hardest existing MDX files (pendulum, kepler, Lagrangian, wave-equation, turbulence).
4. **Physicist + glossary MDX migration.** Write `content/physicists/*.en.mdx` and `content/glossary/*.en.mdx` from current TS data (one-off script is fine). Strip the TS files to structural metadata. (Or pick the alternative route in §Metadata arrays.)
5. **Publish CLI.** `scripts/content/publish.ts`. Dry-run mode. Reports added / updated-by-hash / unchanged / removed rows.
6. **One-topic end-to-end.** Publish English for `classical-mechanics/the-simple-pendulum`. Rewrite its `page.tsx` to read from DB via `getContentEntry()`. Diff rendered HTML before vs after — must match modulo expected structural changes.
7. **Roll out to remaining topics + physicists + glossary.** Mechanical once step 6 proves the renderer. Delete legacy MDX-importing page code.
8. **Translation CLI.** `scripts/content/translate.ts`. Start with Hebrew (existing `.he.mdx` translations become the gold reference for prompt tuning — compare AI output against human-translated existing content, iterate on the prompt until structurally and semantically equivalent). Then extend to whatever locales are next.
9. **Embeddings CLI.** `scripts/content/embed.ts` — embeds each paragraph/equation block into `content_embeddings`. No feature consumes it yet; this just keeps data fresh for the future AI work.
10. **Cleanup.** Delete orphaned `content.{locale}.mdx` files for non-en locales (translations now live in DB). English MDX stays.

Steps 1–6 are the critical path to proving the architecture works on one page; anything after step 6 is incremental rollout plus follow-ups.

## Where to start

**First PR:** steps 1–2 together. Migration + block types + renderer + simulation registry + unit tests. Zero runtime behaviour change (no page yet consumes the renderer), but it gives every later PR a stable foundation and exercises the schema with real fixtures.

## Testing

- **Unit:** block renderer for every block + inline type. Slug resolution in `<PhysicistLink>` / `<Term>`. Simulation allowlist lookup (hit + miss). Inline recursion. Section nesting.
- **Integration — parser:** per-JSX-tag fixture tests (one for each transformer listed in §Publish pipeline). Plus golden-file tests on the five hardest existing MDX files (pendulum, kepler, Lagrangian, wave-equation, turbulence) — parser output vs. committed JSON fixture.
- **End-to-end — one topic:** after step 6, run `pnpm build` and diff rendered HTML of `/en/classical-mechanics/the-simple-pendulum` before vs after migration. Allow differences only in the locale-fallback banner (absent before, absent here too since English is primary); the body must match.
- **Translation pipeline:** fixture test that translates a known English block tree through the LLM prompt and asserts structural equality with the Hebrew block tree (same block types in same order, same slugs, same `tex`, only text fields differ). Second test: asserts no slug resolution errors post-translation.
- **RLS:** anon client can SELECT from `content_entries` + `content_embeddings`, cannot INSERT/UPDATE/DELETE.

## Risks

- **MDX → blocks parser loses a corner case.** Mitigation: golden-file tests on five hardest topics; migration only flips a page to DB after diff is clean.
- **RLS misconfigured, content leaks or 404s.** Mitigation: explicit public-select policy in migration; end-to-end test that fetches as anon.
- **Simulation references go stale** (component renamed in code but not in DB). Mitigation: allowlist lookup throws at render; CI job `pnpm content:validate` reads all DB rows and checks every `component` name still exists in the registry.
- **Translation drift** — English edits without re-translating. Mitigation: `source_hash` column; CI job flags stale locale rows. Site falls back to English automatically.
- **Build-time static generation** reads from Supabase — if Supabase is down at build, build breaks. Mitigation: cache last-good content JSON in the repo under `.content-cache/` (gitignored) and fall back to it if fetch fails.

## Out-of-scope follow-ups (for later specs)

- AI retrieval endpoints (photo-to-lesson) — builds on `content_embeddings`.
- Mobile API (REST or tRPC edge endpoint wrapping `getContentEntry`) — trivial once DB is the source of truth.
- Admin UI for reviewing / editing translations without touching SQL.
- Per-block versioning / change history.
- Community-sourced translation corrections.
