## 1. The crash: SwiftMath's spacing assert

**This was the serious finding.** Twenty live equations — including
`\nabla \times \mathbf{E} \;=\; -\frac{\partial \mathbf{B}}{\partial t}` — did
not fall back. They **aborted the process**:

```
SwiftMath/MTTypesetter.swift:810: Assertion failed:
Invalid space between Relation and Binary Operator
```

Release builds strip `assert`, so shipped code merely rendered the pair with
zero spacing. Debug builds — every developer's app, every `swift test` — died.
A reader scrolling to Faraday's law in a Debug build took the app down.

### Root cause

TeX never leaves a Bin atom next to a Rel/Open/Punct/Bin: it reclassifies it as
Ord, because the inter-element spacing table has no entry for those pairs.
SwiftMath does the same in `MTMathList.finalized` — but it only inspects the
**immediately** preceding atom. In `= \; -b` a *spacing* atom sits in that slot,
and `.space` is not in its `isNotBinaryOperator` set, so the `-` stays Bin.

The typesetter then walks the same list and *does* look past spacing atoms
(`MTTypesetter.createDisplayAtoms`, `case .space:` continues without touching
`lastType`), asks for the spacing between Relation and Binary Operator, and hits
the impossible cell.

`\;=\;` is the house idiom in this content, so any equation whose right-hand
side opens with a sign trips it.

### Fix (in `PXBlocks`)

`PhysicsPackages/Sources/PXBlocks/Formula/MathListSpacing.swift` applies TeX's
own rule to the parsed list, looking past spacing and style atoms exactly as the
typesetter does. Demoting Bin→Unary is lossless: `MTTypesetter.preprocess` turns
every Unary into an Ord anyway, which is the class TeX would have assigned.

Injecting a corrected list means typesetting through `MTMathUILabel` rather than
`MTMathImage` (which takes only a string), and that is a view — hence main
thread. So `TeXRenderer` keeps its off-main fast path for everything and routes
only flagged strings (~1% of the corpus, once per `FormulaKey`, then cached)
through `renderNormalized`. `SpacingTrapTests` pins that both paths agree on
size and baseline to 0.01pt, so a formula's layout never depends on its route.

Verified: the `--hazards` pass re-run against a **Debug** build after the fix
reports **0 of 2352** strings aborting.

### Upstream

Worth filing against `mgriebling/SwiftMath` — the fix is to skip `.space` and
`.style` atoms when `finalized` looks back. Delete `MathListSpacing.swift` if a
release picks it up. (`MTMathTable.finalized` has a separate latent bug in the
same file: `for var row in table.cells` mutates a copy, so its cell rewrites are
discarded. It is harmless today only because each cell is typeset through
`createLineForMathList`, which finalizes again.)

## 2. The 137 failures: all unsupported macros

Every failure is `Invalid command \x` from `MTMathListBuilder`. **No genuine
typos, no malformed input, no environment failures** — SwiftMath simply
implements a narrower macro set than KaTeX.

Each falls back to the designed mono literal (§5) and is logged once. Nothing
renders blank; nothing crashes.

| Macro | Strings | Occ | Fallback verdict | Upstream suggestion (content edit — **not made this session**) |
|---|---:|---:|---|---|
| `\tfrac` | 62 | 68 | acceptable | → `\frac` (or `\textstyle\frac{…}{…}` to keep the size); both parse |
| `\bigl` `\bigr` | 14 | 14 | acceptable | drop the sizer, or `\left(…\right)`; both parse |
| `\boldsymbol` | 9 | 9 | acceptable | → `\mathbf` (loses italic bold on Greek) |
| `\dfrac` | 7 | 7 | acceptable | → `\frac` or `\displaystyle\frac` |
| `\Box` | 7 | 7 | acceptable | → `\square` |
| `\big` | 6 | 6 | acceptable | drop the sizer |
| `\lesssim` | 4 | 4 | **keep the literal** | no SwiftMath equivalent; `\lesssim` reads fine as text. Alternative: `\sim` loses the inequality |
| `\lvert` `\rvert` | 4 | 4 | acceptable | → `\left|…\right|` (parses) |
| `\implies` | 4 | 4 | acceptable | → `\Longrightarrow` |
| `\Bigl` `\Bigr` | 4 | 4 | acceptable | drop the sizer |
| `\mathscr` | 3 | 7 | acceptable | → `\mathcal` (script null infinity `\mathscr{I}^{+}` reads correctly as `\mathcal{I}^{+}`) |

| `\xrightarrow` | 1 | 1 | acceptable | → `\rightarrow` with the label moved into surrounding prose |

Replacement availability was probed against SwiftMath directly, not assumed:
`\frac` `\ldots` `\cdots` `\square` `\Rightarrow` `\Longrightarrow` `\mathcal`
`\mathbf` `\mathrm` `\left(…\right)` `\left|…\right|` `\Vert` `\int\int`
`\dot` `\ddot` `\to` `\rightarrow` `\lim` `\sim` `\approx` `\simeq`
`\textstyle` `\displaystyle` `\bar` all parse. `\lVert`/`\rVert`, `\iiint`,
`\overbrace` and `\lesssim` do not.

**No web content was edited** (P3-5 do-not). The table above is the suggestion
list for a content pass; the four "keep the literal" rows are the ones where the
mono fallback is the better answer regardless.

## 3. Running it

The audit is an SPM executable, `Sources/TeXAudit` (not a test target: it needs
the network, caches a corpus, and writes an artifact — none of which belong in
the fast `swift test` loop).

```bash
cd ios

# Full audit. --refresh re-fetches from PostgREST; without it the cached
# corpus is reused, so triage needs the network exactly once.
# Release, because SwiftMath's asserts otherwise stop the run at the first
# hazard — and Release is what ships.
swift run -c release --package-path PhysicsPackages TeXAudit --refresh

# Crash pass: forks one worker per string against a DEBUG build and records
# any that abort. ~3 min. Annotates the report written above.
swift run --package-path PhysicsPackages TeXAudit --hazards

cat tex-audit-report.json | jq '.totals, (.assertionHazards | length)'
```

The anon key comes from `--anon-key`, else `PX_SUPABASE_ANON_KEY`, else
`NEXT_PUBLIC_SUPABASE_ANON_KEY` in the web repo's `.env.local` (02 §1 — never
committed into the iOS repo). `--fail-on-failures` turns the run into a CI gate
once the content pass above has landed.

Regression cover that does *not* need the network: `SpacingTrapTests` (the
crash class, both render paths) and `FormulaSampleTests` (the Gallery corpus,
which now includes live failures).
